# ContextVault — Summary Layer Ideas

*Documenting ideas for discussion. Not a spec yet.*

---

## The Problem

ContextVault stores full-fidelity Git history. It's complete but not *summarized*. 

When an LLM resumes a session, it needs memory — but:
- Loading all files = expensive, slow
- Loading nothing = no context
- Keyword search = requires knowing what to look for

LLMs need a **"table of contents"** or **"progressive disclosure"** layer on top of Git storage.

---

## Concept: Layered Memory Loading

Think of it like reading a document:

```
Layer 0: Title/Summary — "what happened overall"
Layer 1: Table of Contents — "which files changed and why"
Layer 2: Diff View — "what exactly changed in each file"  
Layer 3: Full Content — "the actual file content"
```

An LLM (or UI) starts at Layer 0 and drills down only when needed. This mirrors how humans read docs.

---

## Idea 1: Auto-Generated Commit Summaries

**What:** Each commit gets a 1-3 sentence TL;DR generated by an LLM.

**How:**
- Hook into the commit creation flow in Commit Gateway
- Send the diff to an LLM (fast model, e.g. GPT-4o-mini or local Ollama)
- Store summary in commit metadata field: `summary: "Added JWT auth to /login, refactored user model, deleted deprecated auth.js"`

**API:**
```
GET /workspaces/:id/summaries?limit=10
→ [{ commitId, summary, filesChanged, agents: ["agent-xyz"], createdAt }]
```

**Pros:**
- Cheap per-commit cost (~$0.01/commit with GPT-4o-mini)
- Stored inline in Git commit — no new storage system
- Enables "what happened since I last checked?"

**Cons:**
- LLM call on every commit = latency on finalization
- Summary quality varies
- Async fallback adds complexity

**Who wants it:** Any agent that resumes and needs a quick status before loading full memory.

---

## Idea 2: Workspace "Digest"

**What:** A rolling summary of a workspace's state — not per-commit, but as of *right now*.

**How:**
- Separate from Git history
- Updated on every commit (or on-demand)
- Shows: current state of key files, active topics, recent changes
- Think: "This workspace is about X. Last 5 changes were Y. Current files are Z."

**Who wants it:** LLMs doing long-running tasks that need to "re-orient" quickly.

---

## Idea 3: Selective / Targeted Retrieval

**What:** Instead of summaries, add a "relevance score" to files based on query.

**How:**
- LLM asks: "what's relevant to the auth module?"
- System returns: files most related to auth, ranked by change frequency + recency
- No vectors — use keyword + graph approach (files changed together are related)

**Who wants it:** Agents working on specific features/tasks.

---

## Idea 4: Progressive UI Loading

**What:** Vault Browser UI gets progressive loading — load summary first, files on demand.

**How:**
- Dashboard shows workspace summary (auto-generated)
- Clicking workspace shows commit summaries + file list (no file content)
- Clicking file loads content
- Reduce initial payload dramatically

**This is a UI feature, not an LLM feature.**

---

## What Makes Sense for Phase 11?

**Lowest lift + highest impact:** Auto-generated commit summaries + `/summaries` endpoint.

Why:
- Hooks into existing flow (commit creation)
- No new storage layer (Git commit metadata is already there)
- Immediate value for LLMs AND human auditors
- Enables "what changed?" at a glance

**Next step after that:** Workspace digest (rolling summary).

**Vector/semantic search is a later phase** — requires separate storage, embedding model, and significantly more complexity.

---

## Open Questions

1. **Sync or async?** LLM call on every commit = sync latency. Should summaries be generated async (queued job)?

2. **Which LLM?** GPT-4o-mini is cheap. Ollama is free but requires local model. Should we support both?

3. **Summary quality?** One LLM-generated sentence might not be useful. Should summaries have a schema? e.g. `{ what: "...", why: "...", impact: "low/medium/high" }`

4. **Retention?** Keep summaries forever or prune old ones? Full history is kept but summaries might not need to be.

5. **Per-customer LLM key?** Generating summaries uses LLM credits. Should customers pay for their own?
